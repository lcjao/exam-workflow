## 一、问题背景

传统 AI 辅助学习的痛点：
- 每次开始都要重新描述学习背景
- AI 不记得上次学到哪里
- 没有系统的进度追踪
- 学完某个知识点不知道下一步该做什么
- 练习题做完没有评分反馈
- 学完就忘，没有复习机制
- 不知道自己什么时候算学完了

**本工作流的解法**：
将所有学习状态、答案、进度都写入本地 Markdown 文件，AI 通过读取文件来获取上下文，用户通过简单命令触发 AI 的下一步行动。

## 二、核心设计决策

### 决策1：用文件代替对话记忆

**原因**：AI 的对话上下文有长度限制，学完一个模块后对话经常被截断。
**解法**：进度追踪、答案、评估结果都写入本地 Markdown 文件，而不是只存在对话中。
**效果**：即使跨会话、跨设备，AI 也能通过读取文件恢复上下文。

### 决策2：Obsidian 文件即答题纸

**原因**：用户在对话框里答题，字数多了不方便整理；另外对话记录不好回溯。
**解法**：所有答题区设计为 `> 答：` 格式写在 Markdown 文件里，用户在 Obsidian 直接填写。
**效果**：答案和题目在一起，方便复习；AI 读取文件时一次性获取所有答案。

### 决策3：量化评估 + 80% 门槛制 + 补强循环

**原因**：学完就放过，掌握度不够；反复卡在同一模块又会打击积极性。
**解法**：
- 每题按评分标准量化打分（见第四章）
- ≥80% 推进，<80% AI 出补强练习
- 第二次无论如何都推进，避免卡关
**效果**：既保证学习质量，又避免学习者沮丧放弃。

### 决策4：极简命令触发

**原因**：学习时用户需要全神贯注，不想花时间想提示词。
**解法**：核心命令统一为 5 个（见第五章命令速查表）。
**效果**：用户几乎不需要思考命令，自然融入学习流程。

### 决策5：渐进式生成文件

**原因**：一次性生成所有文件会耗费大量 token，且很多文件可能用不上。
**解法**：只有当用户到达某个学习阶段时，才生成对应的文件。
**效果**：节省资源，保持灵活性，文件内容也更贴合用户当前的学习状态。

### 决策6：间隔复习调度（v2.0 新增）

**原因**：线性推进导致学完模块 N 后，模块 N-2 的知识点已经遗忘。
**解法**：基于掌握度自动生成复习队列，掌握度越低复习频率越高。
**效果**：对抗遗忘曲线，长期记忆留存率提升。

### 决策7：文件权限分区（v2.0 新增）

**原因**：AI 写入内容和用户写入内容混在一起，容易冲突或误改。
**解法**：每个文件明确标注三个区域——AI 只读区、AI 可写区、用户只写区。
**效果**：避免 AI 误删用户答案，也避免用户误改 AI 评估数据。

## 三、文件类型（精简为 5 + 2 模式）

v1.0 定义了 9 种文件类型，实际使用中认知负担过重。v2.0 精简为：

### 核心文件（最小可用集，3 个就能跑通）

| 类型 | 文件名 | 职责 | 维护方 |
|------|--------|------|--------|
| 课题地图 | `README.md` | 总览全局，模块列表 + 状态 | AI 更新状态 |
| 模块学习卡片 | `模块01.md` … `模块N.md` | 知识输入 + 答题区 | AI 生成内容，用户填写答案 |
| 进度追踪 | `📊学习进度追踪.md` | 项目大脑，记录一切状态 | AI 自动更新 |

### 按需解锁文件（根据学习阶段生成）

| 类型 | 文件名 | 职责 | 触发条件 |
|------|--------|------|----------|
| 薄弱点补讲 | `薄弱点补讲_XXX.md` | 错题本，二次强化 | 评估 <80% |
| 复习调度 | `📅复习调度.md` | 间隔复习计划 | 完成 2+ 模块后 |
| 总复习冲刺 | `总复习_高频考点冲刺.md` | 考前速复 | 所有模块完成 |
| 毕业评估 | `🎓毕业评估.md` | 最终检验 + 毕业判定 | 用户请求毕业 |

### 已移除的文件类型

| v1.0 类型 | 移除原因 | 替代方案 |
|-----------|----------|----------|
| 专项练习 | 与模块内自测题功能重叠 | 模块卡片内增加"综合题"小节 |
| 速查手册 | 与总复习冲刺重叠 | 合并到总复习冲刺的附录 |
| 背一背 | 与模块内"记忆技巧"重叠 | 模块卡片内增加"必背清单" |
| 模拟试卷 | 使用频率低，生成成本高 | 毕业评估中包含模拟测试环节 |

## 四、量化评估规范

### 4.1 评分公式

```
模块掌握度 = Σ(题目得分 × 题目权重) / Σ(题目满分 × 题目权重) × 100%
```

### 4.2 题型权重

| 题型 | 单题满分 | 权重 | 说明 |
|------|----------|------|------|
| 判断题 | 10 分 | 0.5 | 基础认知检测 |
| 选择题 | 10 分 | 0.7 | 理解程度检测 |
| 简答题 | 20 分 | 1.0 | 核心知识点 |
| 分析题 | 30 分 | 1.5 | 应用能力 |
| 论述题 | 40 分 | 2.0 | 综合能力 |

### 4.3 掌握度等级

| 等级 | 分数范围 | 含义 | 行动 |
|------|----------|------|------|
| ✅ 掌握 | 80-100% | 可以进入下一模块 | 推进 |
| ⚠️ 模糊 | 60-79% | 部分理解但有漏洞 | 薄弱点补讲 → 二次评估 |
| ❌ 未掌握 | 0-59% | 理解不足 | 薄弱点补讲 → 二次评估 → 无论如何推进 |

### 4.4 评估输出结构

```
📊 模块XX 评估结果

总体掌握度：XX%（等级：✅/⚠️/❌）

逐题评分：
  Q1 [简答题] 16/20 权重1.0 → 得分率80%
    ✅ 核心概念正确
    ⚠️ 缺少具体数值支撑，建议补充数据

  Q2 [分析题] 18/30 权重1.5 → 得分率60%
    ❌ 分析方向偏差：混淆了X和Y的概念
    📌 改进：重新阅读模块03"X与Y的区别"

加权总分：XX分 / 满分XX分

📌 下一步：[明确可执行的指引]
```

### 4.5 评分原则

1. **先肯定后指出问题**
2. **具体到知识点/题号**，不笼统说「需要加强」
3. **给改进方向**，不只是说「答错了」
4. **评分尺度一致**：同类题型用相同标准
5. **量化优先**：尽量用分数而非主观判断

### 4.6 特殊处理

| 情况 | 处理方式 |
|------|----------|
| 用户未回答某题 | 记 0 分，列入薄弱点，不强迫 |
| 方向对但表述不完整 | 给部分分（50-70%），提示补充什么 |
| 完全正确但有更好表述 | 给满分，补充进阶说法 |
| 用户认为评分有误 | 允许"申诉"，AI 重新评估并说明理由 |

## 五、命令速查表

### 5.1 核心命令（统一版）

| 命令 | 触发动作 | 说明 |
|------|----------|------|
| **「我答完了」** | AI 读取答案 → 量化评估 → 更新进度 | 提交当前模块答案 |
| **「我学完了」** | AI 评估掌握度 → 决定推进/补强 | 完成阅读但未答题时用 |
| **「补答完了」** | AI 二次评估 → 无论如何推进 | 完成薄弱点补强练习 |
| **「开始模块XX」** | 生成指定模块的学习卡片 | 跳级或重学 |
| **「我要毕业」** | 生成毕业评估 → 判定是否达标 | 请求最终检验 |

### 5.2 辅助命令

| 命令 | 触发动作 | 说明 |
|------|----------|------|
| **「跳过这题」** | 标记该题为跳过，不纳入评分 | 用户已掌握或不想答 |
| **「这个我会了」** | 标记该知识点为已掌握，跳过相关题 | 避免重复学习 |
| **「出综合题」** | 生成跨模块综合练习 | 检验知识串联能力 |
| **「申诉」** | AI 重新评估指定题目 | 用户认为评分有误 |
| **「复习」** | 根据复习调度出复习题 | 手动触发复习 |
| **「继续」** | 从上次断点恢复 | 跨会话恢复 |

### 5.3 命令与 v1.0 的映射

| v1.0 命令 | v2.0 命令 | 变更原因 |
|-----------|-----------|----------|
| 「我答完了」 | 「我答完了」 | 保留 |
| 「我学完了」 | 「我学完了」 | 保留 |
| 「补答完了」 | 「补答完了」 | 保留 |

## 六、文件权限分区

每个文件分为三个区域，用 HTML 注释标记：

```markdown
<!-- AI_READ_ONLY -->
# 以下区域 AI 不得修改
- 用户填写的答案（> 答：后面的内容）
- 用户写的自由区内容
<!-- END_AI_READ_ONLY -->

<!-- AI_WRITABLE -->
# 以下区域 AI 可以写入
- 评估结果
- 掌握度数据
- 进度状态
<!-- END_AI_WRITABLE -->

<!-- USER_ONLY -->
# 以下区域只有用户能改
- 学习笔记
- 个人感悟
- 自由区
<!-- END_USER_ONLY -->
```

**原则**：
- AI 永远不修改用户写的答案
- 用户永远不修改 AI 写的评估公式和评分标准
- 进度数据由 AI 自动更新，用户可手动修正

## 七、毕业机制

### 7.1 毕业标准（需全部满足）

1. 所有模块掌握度 ≥ 80%
2. 能用自己的话解释每个核心概念
3. 能举出教材外的例子
4. 能回答反驳性问题（毕业评估中的压轴题）

### 7.2 毕业流程

```
用户说「我要毕业」
       │
       ▼
  AI 检查所有模块掌握度
       │
       ├─ 全部 ≥80% ──→ 生成毕业评估（含综合测试）
       │                      │
       │                 通过 → 生成毕业笔记 → 归档
       │                 不通过 → 列出薄弱模块 → 补强
       │
       └─ 有模块 <80% ──→ 列出未达标模块 → 建议补强
```

### 7.3 毕业笔记结构

```markdown
# 🎓 [科目名称] 毕业笔记

## 核心概念地图
（所有模块的核心概念 + 关联关系）

## 个人收获
（用户在学习过程中写下的关键感悟）

## 行动项
（学完后要做的 3-5 件事）

## 薄弱环节
（即使毕业了仍需注意的知识点）
```

## 八、复习调度机制

### 8.1 复习规则

| 掌握度 | 首次复习 | 二次复习 | 三次复习 |
|--------|----------|----------|----------|
| ✅ 80-100% | 7 天后 | 30 天后 | 90 天后 |
| ⚠️ 60-79% | 3 天后 | 7 天后 | 14 天后 |
| ❌ 0-59% | 1 天后 | 3 天后 | 7 天后 |

### 8.2 复习调度文件格式

```markdown
# 📅 复习调度

## 今日复习（2026-06-05）
- [ ] 模块02 — 薄弱点：X概念（上次掌握度 65%）
- [ ] 模块01 — 常规复习（上次掌握度 85%，7天到期）

##  upcoming
- 2026-06-08: 模块03 首次复习
- 2026-06-12: 模块01 二次复习
```

### 8.3 复习触发

- **自动触发**：每次用户说「我答完了」时，AI 检查是否有到期复习
- **手动触发**：用户说「复习」
- **复习形式**：只出薄弱知识点的题，不出已掌握的题

## 九、最佳实践（含失败案例）

### ✅ 效果好的做法

1. **模块文件内嵌答题区**：`> 答：` 格式放在题目正下方
2. **每个知识点都有记忆技巧**：口诀/类比/联想
3. **进度追踪文件记录易错点**：每次评估后更新
4. **薄弱点补讲精准匹配**：不泛泛重讲
5. **论述题四段式框架**：总起-分析(3段)-总结
6. **量化评分代替主观判断**：减少"感觉掌握了"的误判（v2.0 新增）
7. **复习调度自动化**：不再依赖用户自觉复习（v2.0 新增）

### ⚠️ 踩过的坑

1. **不要一次性生成太多文件**：生成了不用等于浪费
2. **模块不要太长**：超过 9 个知识点就难消化
3. **改错题要规定格式**：「原文→改成」格式
4. **分析题必须要求写数值**：只写「更严重」不如写具体数据
5. **论述题不能只写关键词**：需要展开成完整句子
6. **评估标准不统一导致用户困惑**：同一道题不同会话给了不同分数（v2.0 已修复）
7. **没有复习导致学完就忘**：模块05学完后模块02全忘了（v2.0 已修复）
8. **命令不统一**：README 和设计文档用了不同命令（v2.0 已修复）
